🔄 TRAP #62 — Position State Desync

Files Modified: range_trader.py, universe_trader.py, rsi_trader.py
Critical

The Problem

When a position was manually closed on Zerodha (or via trailing lock), the running strategy process was completely unaware. It kept its internal state as "OPEN", and on the next signal, placed an opposite order that opened a duplicate reverse position (causing triple trades, heavy tax, and losses).

The Solution

  • Scanner loops now perform live database re-validation against order_store.trades_for(today)["open"] on every tick.
  • If a trade was reconciled as closed at the broker, the strategy instantly clears its memory state (st["position"] = None).
  • This shields the account from duplicate, unhedged entries due to manual interventions.
# Inside range_trader.py scanner loop # Re-validate open position against DB open_trades = order_store.trades_for(today)["open"] open_syms = {t["sym"] for t in open_trades} if st["position"] and st["position"] not in open_syms: log.info(f"Position {st['position']} closed externally. Resetting state.") st["position"] = None # Avoid double exit!

🚫 TRAP #64 — Kite Phantom Positions

Files Modified: broker_sync.py
Critical

The Problem

Zerodha Kite's API completely omits contracts with zero trade history (such as cancelled or rejected orders) from its positions() list. Since the app didn't find the contract in the list, it didn't verify order status, resulting in "phantom positions" tracked in the app database that blocked subsequent entries.

The Solution

  • broker_sync.py modified to verify order logs directly if a database-open contract is missing from the API positions list.
  • Queries Kite order history using broker.order_status.
  • If order is confirmed as CANCELLED or REJECTED, it reconciles it as flat and marks it externally_closed immediately.
# Inside broker_sync.py positions check missing_syms = db_open_symbols - api_positions for sym in missing_syms: status = broker.order_status(b_order_id) if status in ["CANCELLED", "REJECTED"]: # Mark as externally closed so phantom position is cleared order_store.mark_closed(sym, tags=["REJECTED_CLEANUP"])

🔥 TRAP #65 — Proactive Feed Cold-Starts

Files Modified: range_trader.py, universe_trader.py
Optimized

The Problem

WebSocket feed subscriptions are capped at 100 concurrent symbols. When starting up, or when a signal was about to fire, dynamic options contracts were subscribed on-the-fly, meaning the tick data was cold (no ticks available) at the exact moment of signal evaluation.

The Solution

  • Watchlist underlying indices and equities are proactively subscribed to dhan_feed at startup.
  • Scanner loops proactively resolve and subscribe the ATM Call & Put contracts dynamically before signals trigger.
  • Ensures ticks are warm and live liquidity filters have immediate data for pre-trade gating.
# Pre-subscribe ATM contracts on scan tick spot = get_live_spot(symbol) atm_strike = round_to_strike(spot) ce_sec, pe_sec = resolve_options(symbol, atm_strike) # Warm up feed subscription before signal fires dhan_feed.add(("NSE_FNO", ce_sec)) dhan_feed.add(("NSE_FNO", pe_sec))

🛡️ TRAP #70 — Global Option Hedging Switch

Files Modified: risk_gate.py, templates/index.html
New Control

The Problem

Auto-hedging buys further OTM options immediately after short entries. However, the hedge order placement was uncontrolled, placing too many orders or generating unnecessary brokerage costs. The user wanted a global switch to toggle between naked positions and hedged positions.

The Solution

  • Added a global Option Hedging dropdown in the RMS tab dashboard (Hedge vs. Naked).
  • Stored under _risk.global.hedge_enabled.
  • When set to Naked Mode, risk_gate.hedge_config() returns 0, None, skipping hedge placement entirely system-wide.

🔒 TRAP #71 — Per-Instrument Trailing Lock

Files Modified: trader_dashboard.py, templates/index.html
New Control

The Problem

The account-level trailing profit lock tracked aggregate daily P&L. When aggregate P&L fell from its peak, the daemon squared off all positions. This cut off other healthy positions because of a drawdown in one single bad position.

The Solution

  • Added a Trailing Mode dropdown in the RMS tab (Aggregate vs. Per-Instrument).
  • Per-Instrument Mode: tracks individual peak P&L for each open position in _pos_peaks.
  • If a position's P&L falls from its personal peak, only that specific position is squared off, keeping other trades active.
