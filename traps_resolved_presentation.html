🛡️ TRAP #53 / #58: Option Hedging Go Brrr...

Files Modified: risk_gate.py, templates/index.html | Fixed by Antigravity AI
Option Safeguard

The Problem 💸

  • Auto-hedging system was a bit too enthusiastic — buying further OTM options immediately on short entry.
  • Hedge order placement was running uncontrolled, throwing orders around like a shopper on Black Friday.
  • User was getting buried in brokerage fees and unnecessary executions! 📉

The Solution 🚀

  • Added a global "Option Hedging" selector in the RMS tab to toggle between Hedge Mode and Naked Mode.
  • If set to Naked Mode, risk_gate.hedge_config() returns 0, None immediately.
  • This bypasses options contract lookup entirely — no hedge orders are sent. Blissful silence! 🤫

🧟‍♂️ TRAP #62: The Zombie Position Desync

Files Modified: range_trader.py, universe_trader.py, rsi_trader.py | Fixed by Claude
Critical

The Problem 🧟

  • You close a position manually on Zerodha because of panic or trailing lock.
  • The strategy process was sleeping at the wheel, still keeping its in-memory state as "OPEN".
  • On the next opposite signal, the strategy fired an exit order that actually opened a brand new opposite position!
  • 1 trade became 3 trades. More taxes, double losses, and high blood pressure! 📈🤬

The Solution 🚀

  • Scanner loops now perform live database checks against order_store.trades_for(today)["open"] on every tick.
  • If a trade was closed at the broker/database, the strategy instantly clears its memory state (st["position"] = None).
  • Prevents accidental ghost entries dead in their tracks. 🛡️
# Inside range_trader.py scanner loop # Re-validate open position against DB open_trades = order_store.trades_for(today)["open"] open_syms = {t["sym"] for t in open_trades} if st["position"] and st["position"] not in open_syms: log.info(f"Position {st['position']} closed externally. Resetting state.") st["position"] = None # Avoid double exit!

🪁 TRAP #64: Kite's Phantom Omission

Files Modified: broker_sync.py | Fixed by Claude
Critical

The Problem 🕳️

  • Zerodha Kite's positions() API completely hides limit orders that were cancelled or rejected (zero traded qty).
  • Since the contract was missing from the positions query, the app couldn't verify its status.
  • It kept tracking them as open "Phantom Positions" in our DB, blocking any fresh trades! 👻

The Solution 🚀

  • Updated broker_sync.py to check the actual order logs directly using broker.order_status for any unmatched open positions.
  • If order is confirmed as CANCELLED or REJECTED, it marks it externally_closed.
  • Cleans the phantom records without double-counting trades. 🧹
# Inside broker_sync.py positions check missing_syms = db_open_symbols - api_positions for sym in missing_syms: status = broker.order_status(b_order_id) if status in ["CANCELLED", "REJECTED"]: # Mark as externally closed so phantom position is cleared order_store.mark_closed(sym, tags=["REJECTED_CLEANUP"])

🥶 TRAP #65: The Cold WebSocket Freezer

Files Modified: range_trader.py, universe_trader.py | Fixed by Claude
Optimized

The Problem 🧊

  • WebSocket feed has a strict limit of 100 concurrent subscriptions.
  • Dynamic option contracts were subscribed only at the exact millisecond a signal fired.
  • The feed was cold (zero ticks accumulated), causing the live liquidity checks to fail and miss the trade! 🥶

The Solution 🚀

  • Watchlist underlying stocks and indices are subscribed to dhan_feed right at startup.
  • The scanner loop proactively resolves and subscribes likely ATM CE/PE option contracts during each scan iteration.
  • Ticks are warm and warm liquidity data is ready before any signal fires! 🔥
# Pre-subscribe ATM contracts on scan tick spot = get_live_spot(symbol) atm_strike = round_to_strike(spot) ce_sec, pe_sec = resolve_options(symbol, atm_strike) # Warm up feed subscription before signal fires dhan_feed.add(("NSE_FNO", ce_sec)) dhan_feed.add(("NSE_FNO", pe_sec))

🔒 TRAP #71: The "Save Everyone" vs "Save Yourself" Lock

Files Modified: trader_dashboard.py, templates/index.html | Fixed by Antigravity AI
New Control

The Problem 💥

  • The trailing profit lock worked at the total aggregate portfolio level.
  • If P&L dropped from its peak, the daemon hit the panic button and squared off every open trade.
  • This was like a bad team manager: one player performs badly, and the entire team gets fired! NIFTY was forced to close just because SBIN dipped. 🤦‍♂️

The Solution 🚀

  • Added a Trailing Mode select element in the RMS tab to choose between Aggregate and Per-Instrument locks.
  • Per-Instrument Mode: Tracks peak P&L for each open position individually in _pos_peaks.
  • If a single position drops from its peak, only that leg is squared off. The healthy positions are left to keep printing cash! 💰🧗‍♂️
